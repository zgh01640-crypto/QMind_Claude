# 套定额程序开发规划 v3（基于 bs2024_* + boq_* 整合）

## Context

现有 `/boq` 工程管理模块已完整实现：项目列表、SSE 流式匹配、历史批次保存。但它对接的是旧的 `quota_items` 表（旧定额库）。

本次目标：在现有 boq 流程基础上，新增对 `bs2024_subitems`（深圳建筑消耗量标准2024）的套定额支持，整合到新的"新工程管理"菜单入口。

**不重复建表**：复用 `boq_projects` / `boq_sections` / `boq_items` 作为清单数据源，新增 `bs2024_match_runs` + `bs2024_quota_matches` 两张结果表（结构参照现有 boq_match_runs/boq_quota_matches，FK 指向 bs2024_subitems）。

---

# 一、数据来源

## 清单项（输入）
- 表：`boq_projects` → `boq_sections` → `boq_items`
- 字段：`item_code`, `item_name`, `item_description`（项目特征）, `unit`, `quantity`
- 现有 API：`GET /api/boq/projects`, `/api/boq/sections?project_id=X`, `/api/boq/items?project_id=X&section_id=Y`

## 定额库（匹配目标）
- 表：`bs2024_chapters` → `bs2024_subitems`（含 `name_path_json` JSONB）
- 专业说明/规则：`bs2024_sections`（`section_type='intro'/'rules'`，`content_md`）
- 现有 API：`GET /api/building-standard-2024/tree?document_id=X`

## 匹配结果（新建）

### 新增 SQL（`db/schema_bs2024_match.sql`）

```sql
CREATE TABLE IF NOT EXISTS bs2024_match_runs (
    id            SERIAL PRIMARY KEY,
    project_id    INT NOT NULL REFERENCES boq_projects(id) ON DELETE CASCADE,
    chapter_id    INT NOT NULL REFERENCES bs2024_chapters(id),  -- 对应专业
    chapter_name  VARCHAR(200),
    run_name      VARCHAR(200),
    status        VARCHAR(20) DEFAULT 'running',
    total_items   INT DEFAULT 0,
    matched_items INT DEFAULT 0,
    created_at    TIMESTAMP DEFAULT NOW(),
    finished_at   TIMESTAMP
);

CREATE TABLE IF NOT EXISTS bs2024_quota_matches (
    id              SERIAL PRIMARY KEY,
    run_id          INT NOT NULL REFERENCES bs2024_match_runs(id) ON DELETE CASCADE,
    boq_item_id     INT NOT NULL REFERENCES boq_items(id) ON DELETE CASCADE,
    subitem_id      INT NOT NULL REFERENCES bs2024_subitems(id),
    subitem_code    VARCHAR(50),
    work_procedure  VARCHAR(200),     -- 所属工序名称
    qty_factor      NUMERIC(10,4) DEFAULT 1.0,
    factor_explanation TEXT,          -- 换算说明（如 m³→10m³ 系数0.1）
    ai_reasoning    TEXT,             -- 完整推理过程
    confidence      NUMERIC(4,2),     -- 0.00~1.00
    status          VARCHAR(20) DEFAULT 'ai',  -- ai/confirmed/rejected
    created_at      TIMESTAMP DEFAULT NOW(),
    UNIQUE (run_id, boq_item_id, subitem_id)
);
```

# 四、大模型提示词设计

## System Prompt（含专业定额上下文，启用 prompt cache）

```
你是一名深圳市建筑工程造价工程师，精通《深圳市建筑工程消耗量标准》（SJG 171-2024）。

## 本次专业定额：{chapter_name}

### 专业说明
{intro_content_md}

### 工程量计算规则
{rules_content_md}

### 可用定额子目列表
[code] 路径（>分隔） | 单位 | 全费用单价 | 工作内容
{每条子目一行}

---（以上内容缓存）---
```

`cache_control: {"type": "ephemeral"}` 标记在 system 消息的最后一个 content block。

## Step 1 User Prompt（特征解析 → 工序）

```
请分析清单项，拆解为本专业范围内的工序列表。

清单项：
  名称：{item_name}
  项目特征：{item_description}
  单位：{unit}，工程量：{quantity}

输出 JSON：
[{"seq":1,"name":"工序名","description":"含规格参数的具体说明"},...]
只列本专业（{chapter_name}）工序，其他专业不列入。
```

## Step 2 User Prompt（工序 → 子目）

```
请为每条工序从上方子目列表中找最匹配的子目。

工序列表：
{seq}. {name}：{description}
...

对每条工序输出（工具调用 submit_matches）：
- subitem_code：必须来自上方列表，不能编造
- work_procedure：工序名
- qty_factor：换算系数（清单量 / 定额量，如 m³→10m³ 则 0.1）
- factor_explanation：换算说明
- confidence：0.0~1.0
```

使用 **Claude Tool Use**（`submit_matches` 工具）强制结构化输出：
```python
tools = [{
    "name": "submit_matches",
    "description": "提交工序与定额子目的匹配结果",
    "input_schema": {
        "type": "object",
        "properties": {
            "matches": {
                "type": "array",
                "items": {
                    "type": "object",
                    "properties": {
                        "subitem_code": {"type": "string"},
                        "work_procedure": {"type": "string"},
                        "qty_factor": {"type": "number"},
                        "factor_explanation": {"type": "string"},
                        "confidence": {"type": "number"}
                    },
                    "required": ["subitem_code", "work_procedure", "qty_factor", "confidence"]
                }
            }
        },
        "required": ["matches"]
    }
}]
```

# 五、新增 API（`api/routers/bs2024_match.py`）

```
GET  /bs2024-match/chapters                    可用专业列表
GET  /bs2024-match/chapters/{id}/context       专业 system prompt 文本预览
POST /bs2024-match/runs/stream                 批量套定额（SSE 流式）
GET  /bs2024-match/runs?project_id=X           工程历史批次列表
GET  /bs2024-match/runs/{run_id}/matches       批次匹配结果
PUT  /bs2024-match/matches/{id}               确认/拒绝一条匹配
```

### SSE 事件类型（与现有 boq 流式保持一致）
```
run_start   → {run_id, total}
item_start  → {index, total, boq_item_id, item_name}
reasoning_token → {token}          # 流式推理字符
item_done   → {boq_item_id, matches:[{subitem_code,work_procedure,confidence,qty_factor,...}]}
run_done    → {run_id, total, matched}
run_error   → {error}
```

在 `api/main.py` 注册：
```python
from api.routers import bs2024_match
app.include_router(bs2024_match.router, prefix="/api", tags=["bs2024-match"])
```

# 六、实施顺序

1. **数据库**：创建 `db/schema_bs2024_match.sql`，在 Docker 容器中执行
2. **后端**：编写 `api/routers/bs2024_match.py`
   - `build_system_prompt(chapter_id)` → 查 intro/rules content_md + 子目列表
   - `stream_match_item(boq_item, system_prompt)` → 两步 Claude 调用，yield SSE 事件
   - SSE 端点：`POST /bs2024-match/runs/stream`（body: project_id, chapter_id, item_ids, run_name）
3. **注册路由**：`api/main.py` 添加 bs2024_match router
4. **前端工程列表**：`web/app/new-boq/page.tsx`（复用现有 boq projects API）
5. **前端工程详情**：`web/app/new-boq/[id]/page.tsx`（选专业 + 清单多选 + SSE 流式面板）
6. **前端结果详情**：`web/app/new-boq/[id]/run/[rid]/page.tsx`
7. **Navbar**：添加"新工程管理"菜单项
8. **测试**：砌筑工程小样本（砖基础 m³ 50，选第1章）

# 七、验证方法

1. 执行 schema SQL，确认两张新表创建成功
2. 启动 API（`uvicorn api.main:app --port 8004`），调用 `GET /api/bs2024-match/chapters` 返回 5 个专业
3. 在"新工程管理"选一个已有工程，选定额专业=砌筑工程，输入批次名，点"开始套定额"
4. 检查 SSE 流：`run_start` → 多条 `item_start/reasoning_token/item_done` → `run_done`
5. 跳转结果详情页，确认每条清单项有对应子目编码、换算系数、置信度
6. 验证 `bs2024_match_runs.status='done'` 且 `bs2024_quota_matches` 有记录

**测试样本**：
```
清单项：砖基础
项目特征：砖品种：MU10烧结普通砖 240×115×53；砂浆：M10混合砂浆；部位：砖基础
单位：m³  工程量：50
```
期望：工序=砌砖体+砂浆制作；子目来自 bs2024_subitems 第1章；m³→10m³ 系数0.1
